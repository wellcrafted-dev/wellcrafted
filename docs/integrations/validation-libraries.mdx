# Validate branded values

`Brand<T>` is type-only. It distinguishes values at compile time but performs no parsing, runtime validation, or serialization work. Pair the type with a validator at the boundary where an unknown or plain value becomes trusted domain data.

```typescript
import type { Brand } from "wellcrafted/brand";

type FileId = string & Brand<"FileId">;
```

The validator checks the runtime value first. Its final transform is the one place that asserts the branded output.

<Tabs>
  <Tab title="ArkType">
    ```typescript
    import { type } from "arktype";
    import type { Brand } from "wellcrafted/brand";

    type FileId = string & Brand<"FileId">;
    const FileId = type("string")
      .narrow((value, ctx) =>
        value.length > 0 ? true : ctx.mustBe("a non-empty file id"),
      )
      .pipe((value): FileId => value as FileId);
    ```
  </Tab>
  <Tab title="Zod">
    ```typescript
    import { z } from "zod";
    import type { Brand } from "wellcrafted/brand";

    type FileId = string & Brand<"FileId">;
    const FileId = z
      .string()
      .min(1)
      .transform((value): FileId => value as FileId);
    ```
  </Tab>
  <Tab title="Valibot">
    ```typescript
    import * as v from "valibot";
    import type { Brand } from "wellcrafted/brand";

    type FileId = string & Brand<"FileId">;
    const FileId = v.pipe(
      v.string(),
      v.minLength(1),
      v.transform((value): FileId => value as FileId),
    );
    ```
  </Tab>
</Tabs>

TypeScript lets the type and validator share `FileId` because types and values occupy separate namespaces. In a type annotation it means the brand; at runtime it means the validator.

## Add domain validation before branding

The cast does not validate anything by itself. Put every runtime condition before the transform.

<Tabs>
  <Tab title="ArkType">
    ```typescript
    type Email = string & Brand<"Email">;

    const Email = type("string")
      .narrow((value, ctx) =>
        /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value)
          ? true
          : ctx.mustBe("a valid email address"),
      )
      .pipe((value): Email => value as Email);
    ```
  </Tab>
  <Tab title="Zod">
    ```typescript
    type Email = string & Brand<"Email">;

    const Email = z
      .string()
      .email()
      .transform((value): Email => value as Email);
    ```
  </Tab>
  <Tab title="Valibot">
    ```typescript
    type Email = string & Brand<"Email">;

    const Email = v.pipe(
      v.string(),
      v.email(),
      v.transform((value): Email => value as Email),
    );
    ```
  </Tab>
</Tabs>

Only values produced by the successful validator path carry the runtime-validation evidence your application expects. A direct `value as Email` elsewhere can bypass that boundary, so keep assertions inside constructors or schemas.

Branding does not change the serialized value. A branded string still crosses JSON as a string, and the receiver must validate it again before treating it as branded.

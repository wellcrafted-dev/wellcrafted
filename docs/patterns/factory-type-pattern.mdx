# The `InferError` Pattern for TypeScript Factories

I was building error factories in wellcrafted and kept running into this annoying pattern:

```typescript
const errors = defineErrors({
  ApiError: ({ message }: { message: string }) => ({ message }),
});

// Later, when I need the type...
type ApiError = ReturnType<typeof errors.ApiError>;
// Wait, that's verbose. There must be a better way...
```

Every time I needed to extract the type from an error factory, I'd have to do this awkward dance with `ReturnType` and property access. It felt like I was fighting TypeScript instead of working with it.

## The Problem

Here's what I had:

```typescript
const errors = defineErrors({
  ApiError: ({ message }: { message: string }) => ({ message }),
});

// Using it
const { ApiError, ApiErr } = errors;

// But how do I get the ApiError type cleanly?
// This works but is verbose:
type ApiError = ReturnType<typeof errors.ApiError>;
```

The type exists. TypeScript knows about it. But extracting it requires this verbose, indirect approach.

## The Solution: InferError

So I added an `InferError` utility type:

```typescript
import { defineErrors, type InferError } from 'wellcrafted/error';

const errors = defineErrors({
  ApiError: ({ message }: { message: string }) => ({ message }),
});

type ApiError = InferError<typeof errors, 'ApiError'>;  // That's it!

// Still works with destructuring
const { ApiError, ApiErr } = errors;
```

## How It Works

`InferError` is a conditional type that extracts the return type from a specific factory in a `defineErrors` return:

```typescript
type InferError<T, K extends string> = K extends keyof T
  ? T[K] extends (...args: any[]) => infer R
    ? R
    : never
  : never;
```

It looks up the factory by name, then extracts its return type. Clean, readable, and type-safe.

## Other Places This Pattern Shows Up

Once I noticed this pattern, I started seeing it everywhere:

```typescript
// Zod schemas
const userSchema = z.object({
  name: z.string(),
  age: z.number(),
});
type User = z.infer<typeof userSchema>;

// tRPC
const appRouter = router({
  getUser: procedure.query(() => ({ name: "Alice" })),
});
type AppRouter = typeof appRouter._def;  // Internal type marker

// Custom query builders
const query = createQuery("users");
type QueryResult = typeof query._result;  // Another phantom property
```

These libraries all solved the same problem: How do you give developers a clean way to extract types from runtime values?

## The Lesson

The `InferError` pattern is about making TypeScript ergonomic. Instead of requiring users to understand the internal structure of your factory returns and chain together `ReturnType<typeof ...>` expressions, you provide a clean utility type that does the heavy lifting.

It's just another tool for making TypeScript more ergonomic. And that's what good library design is about: making the right thing easy to do.

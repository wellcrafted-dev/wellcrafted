# Prompt 1: Review and Improve `taggedError` Function Naming

## Context

We have a function in `src/error/utils.ts` called `taggedError` that:
- Takes a string parameter (error name ending in "Error", e.g., `'NetworkError'`)
- Optionally takes generic type parameters for context shape and cause type
- Returns an object with two factory functions:
  - `[Name]`: Creates plain error objects (e.g., `NetworkError`)
  - `[Name with 'Err' suffix]`: Creates `Err<TaggedError>` results (e.g., `NetworkErr`)

Current usage:
```typescript
// Simple
const { NetworkError, NetworkErr } = taggedError('NetworkError');

// With fixed context
const { BlobError, BlobErr } = taggedError<'BlobError', BlobContext>('BlobError');

// With fixed context and cause
const { ApiError, ApiErr } = taggedError<'ApiError', ApiContext, NetworkError>('ApiError');
```

## Your Task

**Spin up a dedicated subagent** to think deeply about naming from the user's perspective.

The subagent should consider:

1. **What the user is thinking when they use this function:**
   - "I want to create an error type called X"
   - "I need factories to build errors of type X"
   - "I'm defining a new error for my application"

2. **What the function actually does:**
   - Takes a string â†’ returns two factory functions
   - The string becomes the `name` property on created errors
   - The factories create plain objects (not class instances)

3. **Naming candidates to evaluate:**
   - `taggedError` (current) - emphasizes "tagged" discriminated union aspect
   - `createTaggedError` (deprecated alias) - explicit "create" verb
   - `defineError` - implies defining a new error type
   - `errorFactory` - describes what it returns
   - `makeError` - simple verb
   - `error` - ultra-minimal (like `z.string()` in Zod)
   - Something else entirely?

4. **Considerations:**
   - How does it read in actual code?
   - Is it clear what the function does?
   - Does it follow TypeScript/JS ecosystem conventions?
   - Is it too long or too short?
   - Does "tagged" add value or confusion for users unfamiliar with the term?

5. **The function signature context:**
   ```typescript
   function taggedError<TName, TContext?, TCause?>(name: TName): {
     [TName]: ErrorConstructor;
     [TName.replace('Error', 'Err')]: ErrConstructor;
   }
   ```

## Expected Output

The subagent should return:
1. A recommended name with strong reasoning
2. Alternative names considered and why they were rejected
3. Any suggestions for improving the API beyond just naming
